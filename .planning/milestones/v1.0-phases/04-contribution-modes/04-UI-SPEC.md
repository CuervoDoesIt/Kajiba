# Phase 4 -- UI Design Contract

> Visual and interaction contract for CLI terminal output. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> This phase is a CLI-only Python project using Rich for terminal rendering and Click for command structure. There is no browser-based frontend. All "UI" contracts below specify Rich markup styles, panel layouts, table structures, and prompt patterns.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (CLI-only -- no web component library) |
| Preset | not applicable |
| Component library | Rich (panels, tables, text styling) + Click (prompts, choices) |
| Icon library | not applicable (terminal output) |
| Font | System terminal font (monospace, user-controlled) |

---

## Spacing Scale

Terminal spacing is controlled by blank lines and indentation, not pixel values. The scale below uses character units consistent with the existing CLI patterns.

| Token | Value | Usage |
|-------|-------|-------|
| inline | 0 lines | Between related fields within a panel or table row |
| tight | 1 blank line | Between elements within a single command's output (e.g., after a panel, before a prompt) |
| section | 2 blank lines | Between major output sections (e.g., record preview vs. action prompt) |
| indent-1 | 2 spaces | Sub-items under a heading (existing pattern: `"  Staging path: ..."`) |
| indent-2 | 4 spaces | Nested sub-items (not currently used; reserved) |

Exceptions: Rich `Table` and `Panel` components manage their own internal padding. Do not add manual spacing inside Rich components.

## Typography

Terminal output is monospace; "typography" maps to Rich text styles.

| Role | Rich Style | Usage |
|------|------------|-------|
| Body | (no style / default) | Standard output text, record field values |
| Label | `[bold]` | Field names in panels and key-value output (e.g., `[bold]Quality Tier:[/bold]`) |
| Heading | `[bold]` on `console.print()` | Section headings (e.g., `[bold]Staged records:[/bold]`) |
| Dim | `[dim]` | Secondary info, deferred values, activity summaries, skip confirmations |

Weights used: exactly 2 -- default (regular terminal weight) and `bold`.

## Color

Rich named colors used across CLI output. These follow the established palette from existing commands.

| Role | Rich Color | Hex Approx | Usage |
|------|------------|------------|-------|
| Dominant (60%) | (default terminal fg/bg) | user theme | All body text, tables, panels |
| Secondary (30%) | `dim` | -- | Metadata, file paths, non-critical info, activity summaries |
| Success | `green` | -- | Approved actions, successful operations (e.g., `"Record submitted"`, `"Approved and submitted"`) |
| Warning | `yellow` | -- | Empty states, flagged items, skipped records, cancelled actions |
| Error / Destructive | `red` | -- | Rejected records, invalid config values, errors |
| Accent: quality tiers | see tier map | -- | Quality tier labels only |

### Quality Tier Color Map (existing, reuse exactly)

| Tier | Rich Color |
|------|------------|
| gold | `yellow` |
| silver | `white` |
| bronze | `dark_orange` |
| review_needed | `red` |

### Accent Reserved For

Accent colors are reserved for specific semantic meanings. Never use `green` for non-success output, `yellow` for non-warning output, or `red` for non-error output.

- `green` -- approve/submit/export success confirmations only
- `yellow` -- warnings, empty states, flagged-for-review items, quality tier "gold" only
- `red` -- errors, destructive confirmations, quality tier "review_needed" only
- `bold cyan` -- table column headers for field names (existing pattern in preview table)
- `dark_orange` -- quality tier "bronze" only

## Copywriting Contract

### Review Command (`kajiba review`)

| Element | Copy |
|---------|------|
| Primary CTA prompt | `Action` (with choices: approve, reject, skip, quit) |
| Default action | `skip` (safe default -- no data moved or deleted) |
| Empty state heading | `No records in staging to review.` |
| Empty state body | (none needed -- staging path is not shown for review; user knows where staging is from other commands) |
| Approve success | `Approved and submitted: {outbox_file.name}` (green) |
| Reject confirmation | `Record rejected and removed from staging.` (red) |
| Skip confirmation | `Skipped.` (dim) |
| Review summary | `Review complete: {N} approved, {N} rejected, {N} skipped` (bold) |
| Record label | `Reviewing: {filepath.name}` (bold) |

### Review Destructive Action: Reject

| Element | Copy |
|---------|------|
| Action name | `reject` |
| Behavior | Permanently removes the record from staging. No confirmation prompt beyond the initial choice. The 4-way choice (approve/reject/skip/quit) makes the action explicit. |
| Rationale | Adding a separate "Are you sure?" for reject would break the one-at-a-time review flow rhythm. The user typed "reject" deliberately from a 4-option prompt. |

### Config Subcommands

| Element | Copy |
|---------|------|
| `config set` success | `Set {key} = {value}` (green) |
| `config set` invalid key | `Unknown config key: {key}` (red), followed by `Valid keys: {comma-separated list}` (dim) |
| `config set` invalid value | `Invalid value for {key}: {value}` (red), followed by `Valid options: {comma-separated choices}` (dim) |
| `config get` output | `{key} = {value}` (no color, plain text) |
| `config get` unknown key | `Unknown config key: {key}` (red) |
| `config get` unset key | `{key} = {default} (default)` (dim for the "(default)" suffix) |
| `config show` | Reuses existing config table format: `Table(title="Kajiba Configuration")` with "Setting" (bold) and "Value" columns. Add "Source" column showing `config` or `default`. |
| PyYAML missing for `set` | `PyYAML is required for config set. Install it: pip install pyyaml` (raised as ClickException) |

### Activity Notification (D-10)

| Element | Copy |
|---------|------|
| Auto-submitted only | `{N} record(s) auto-submitted` (dim) |
| Queued only | `{N} queued for review` (dim) |
| Both | `{N} record(s) auto-submitted, {N} queued for review` (dim) |
| Placement | First line of output, before any command-specific content. Single line, dim style. Appears once then clears. |

### Continuous Mode Logging (silent, in activity log)

| Element | Format |
|---------|--------|
| Auto-submit entry | `{"action": "auto_submitted", "record_id": "{id}", "quality_tier": "{tier}", "timestamp": "{ISO8601}"}` |
| Queued entry | `{"action": "queued_for_review", "record_id": "{id}", "quality_tier": "{tier}", "timestamp": "{ISO8601}"}` |

## Component Inventory

This phase introduces 3 new CLI output patterns and modifies 1 existing pattern.

### New: Review Record Display

Reuses `_render_preview()` exactly as-is. The review command calls the same preview infrastructure used by `kajiba preview`. No new rendering code.

**Structure:**
1. Record label: `console.print(f"\n[bold]Reviewing:[/bold] {filepath.name}")`
2. Full preview via `_render_preview(preview_record, quality_dict, scrub_stats, flagged_items=all_flagged)`
3. Action prompt: `click.prompt("Action", type=click.Choice(["approve", "reject", "skip", "quit"]), default="skip")`
4. Result message (green/red/dim depending on action)
5. After all records: summary line

### New: Activity Notification Banner

**Structure:** Single `console.print()` call with dim style. Placed in the CLI group callback (runs before every command).

```
console.print(f"[dim]{', '.join(parts)}[/dim]")
```

No panel, no table, no box. A single dim line that reads naturally: "2 record(s) auto-submitted, 1 queued for review".

### New: Config Set/Get Output

**`config set` success:**
```
console.print(f"[green]Set {key} = {value}[/green]")
```

**`config get` output:**
```
console.print(f"{key} = {value}")
```
Or with default indicator:
```
console.print(f"{key} = {value} [dim](default)[/dim]")
```

### Modified: Config Show Table

Existing `config` command body moves to `config show`. Add a "Source" column:

```python
table = Table(title="Kajiba Configuration")
table.add_column("Setting", style="bold")
table.add_column("Value")
table.add_column("Source", style="dim")
```

Source values: `"config"` when read from `~/.hermes/config.yaml`, `"default"` when using hardcoded default.

## Interaction Patterns

### Review Flow (One-at-a-Time)

```
$ kajiba review
2 record(s) auto-submitted              <-- activity notification (dim, if pending)

Reviewing: session_abc123.json          <-- record label (bold)

[Full preview panel output]             <-- reused from _render_preview()
[Quality & Annotations panel]
[Scrubbing Summary table]

Action (approve, reject, skip, quit) [skip]: approve    <-- Click prompt
Approved and submitted: record_def456.jsonl              <-- success (green)

Reviewing: session_xyz789.json
...

Review complete: 2 approved, 0 rejected, 1 skipped      <-- summary (bold)
```

### Config Set Flow

```
$ kajiba config set contribution_mode continuous
Set contribution_mode = continuous

$ kajiba config set min_quality_tier platinum
Error: Invalid value for min_quality_tier: platinum
Valid options: gold, silver, bronze

$ kajiba config set unknown_key foo
Error: Unknown config key: unknown_key
Valid keys: contribution_mode, min_quality_tier, consent_level, auto_submit_interval, ...
```

### Config Get Flow

```
$ kajiba config get contribution_mode
contribution_mode = continuous

$ kajiba config get min_quality_tier
min_quality_tier = silver (default)
```

### Config Show Flow

```
$ kajiba config show
```
